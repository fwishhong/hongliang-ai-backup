## 一、引题

如果地府也开始裁员，恶鬼暴动，奈何桥灯快要熄灭，新任阴阳司主事临时上岗，会发生什么？这就是《地府小围城》的起点。它是一款 Q 版可爱地府风横屏塔防小游戏，玩家在鬼门关到奈何桥灯/轮回井的道路两侧建塔、清障、抵御恶鬼，守住轮回秩序。

但这篇文章的重点不只是分享一款游戏，而是分享一套可复制的 AI Coding 工作流：

- 怎么用 Harness 把需求、设计、架构、安全、质量和执行计划组织成一个项目工程
- 怎么用 Codex 在真实代码仓库里持续写代码、改文档、跑验证
- 怎么用 image2 生成能真正接入游戏的背景、Boss 和 UI 素材

### AI Coding 工作流总览

| 环节 | 使用工具 | 目标 |
|------|----------|------|
| Harness 工程 | Harness + Codex | 用文档体系管理需求、设计、架构、安全、质量、计划和验收 |
| 游戏策略 | 人 + Codex | 把脑洞变成核心循环、关卡节奏、塔/怪/Boss 规则 |
| 素材生产 | image2 + Codex | 生成背景、角色、Boss、技能图标，并用素材规范筛选方案 |
| 架构方案 | Codex | 生成架构方案、设计方案、运行时边界和平台策略 |
| 任务生成 | Codex | 把规格拆成 progress.md 里的 P0-P4 任务线 |
| 任务执行 | Codex + npm scripts | 写代码、改配置、跑构建、跑模拟、更新进度 |
| 人工验收 | 人 | 判断好不好玩、能不能看懂、是否值得继续迭代 |

### "工业级可玩"的定义

至少做到五点：
1. 代码有边界
2. 素材可维护
3. 关卡可扩展
4. 构建可重复
5. 玩家能真正完成一局并理解为什么赢或输

## 二、Harness 项目框架搭建

这里要先纠正一个容易误解的点：**Harness 不是游戏本身的代码框架**。《地府小围城》的游戏代码框架是 TypeScript + Canvas 2D + 自研轻量运行时，代码在 src/、content/、assets/、scripts/ 等目录里。

**Harness 是另一层东西**：它是围绕项目建立的一整套需求、设计、架构、安全、质量、执行计划和验收文档工程，用来让 Codex 理解项目、约束输出、持续推进任务。

很多 AI Coding 项目失败，不是因为 AI 不会写代码，而是因为它没有稳定上下文。今天让它写 UI，明天让它改资源，后天让它调数值，如果没有统一的规格和验收，它很容易每次都像重新接手一个陌生项目。

所以**第一步不是直接让 Codex 写游戏，而是先建立 Harness 项目工程**：

```
game-tower/
├── AGENTS.md                    # 文档目录和 ARCHITECTURE.md # 整体架构设计
├── docs/
│   ├── design-docs/             # 设计文档：解决怎么做的问题
│   │   ├── index.md             # 方案设计目标
│   │   ├── core-beliefs.md      # 核心设计信念
│   │   └── ...
│   ├── exec-plans/              # 任务文件夹
│   │   ├── active/              # 活动中的任务
│   │   ├── completed/           # 已完成任务
│   │   └── tech-debt-tracker.md # 技术债务文档
│   ├── generated/               # 技术产生的文件夹
│   │   └── db-schema.md         # 数据库文件
│   ├── product-specs/           # 需求文档：解决做什么的问题
│   │   ├── index.md             # 需求目录
│   │   ├── new-user-onboarding.md # 新用户旅程需求
│   │   └── ...
│   └── references/               # 关联文档
│       ├── design-system-reference-llms.txt
│       ├── nixpacks-llms.txt
│       ├── uv-llms.txt
│       └── ...
├── DESIGN.md                    # 方案设计入口和基本原则
├── FRONTEND.md                  # 前端形态设计
├── PLANS.md                     # 总体计划及计划入口
├── PRODUCT_SENSE.md             # 需求文档入口及基本原则
├── QUALITY_SCORE.md             # 质量文档入口及基本原则
├── RELIABILITY.md              # 可靠性文档入口及基本原则
└── SECURITY.md                 # 安全性文档入口及基本原则
```

### Harness 各模块作用

| Harness 文件/目录 | 作用 |
|------------------|------|
| AGENTS.md | 给 Codex 的硬性协作规则、常用命令、关键代码路径和验收规则 |
| ARCHITECTURE.md | 说明游戏代码分层、平台入口、状态边界、资源加载和构建关系 |
| docs/product-specs/ | 记录游戏目标、核心循环、关卡规划、分享方案、新手引导 |
| docs/design-docs/ | 记录运行时设计、关卡配置、美术规范、Boss 设计和微信技术方案 |
| docs/exec-plans/ | 把规格拆成任务，记录 active、completed、技术债和验收清单 |
| docs/exec-plans/active/progress.md | 记录真实研发顺序、完成项、验证结果、当前限制和归档记录 |
| docs/QUALITY_SCORE.md | 约束质量标准，避免只追求"能跑" |
| docs/RELIABILITY.md | 约束错误处理、资源失败、恢复和稳定性 |
| docs/SECURITY.md | 约束安全边界，尤其是分享 query、云端资源和平台能力 |

有了 Harness，Codex 才不是在空地上随便写代码，而是在一套已有边界里补能力。它知道哪些文件能改、哪些命令要跑、哪些规则不能破坏、哪些需求已经完成、哪些任务还在 active。

### 给 Codex 做 Harness 类任务时的四个必答问题

1. 这个需求对应哪份 product-spec 或 design-doc？
2. 当前 exec-plan 是否已有任务和验收标准？
3. 如果需要改代码，它属于状态层、渲染层、引擎层、平台层还是 content 配置？
4. 改完以后应该跑哪些验证命令，并更新哪些进度文档？

## 三、游戏策略

游戏策略阶段不是堆功能，而是先明确玩家每局到底在做什么。

### 核心循环

1. 进入关卡，观察路线、塔位、障碍和终点
2. 用初始金币建造第一批炮塔
3. 恶鬼从鬼门关出发，沿固定路线推进
4. 炮塔攻击恶鬼，玩家获得金币和阴德
5. 玩家在防守、升级、清障和释放技能之间分配资源
6. 每波结束后准备下一波，也可以提前开波
7. 漏到终点的恶鬼会触发轮回，变强后从起点再跑一圈
8. 全部波次结束后，根据表现结算功德牌，并解锁下一关

### 三个核心策略判断

**第一，塔防的爽点不是"怪越多越好"，而是玩家能读懂每种压力。**
- 第 1 关只教固定路线、建塔、升级、清障和轮回
- 第 2 关再引入饿死鬼和减速塔
- 第 3 关才上第一个 Boss 压力

**第二，关卡必须数据驱动。**
普通关卡优先通过配置扩展，只有新增通用机制、敌人行为、炮塔行为、地图结构或结算规则时才写代码。这样 18 关战役就可以围绕配置持续迭代，而不是每一关都写一堆特例。

**第三，地府题材要变成可玩的机制，而不是只停留在皮肤上：**

- **轮回路**：漏怪不会马上消失，而是回到入口并变强
- **阴德**：击杀敌人掉落阴德，自动入账，用于主动技能
- **招魂幡**：第 4 关解锁，全屏牵回危险敌人并减速
- **阎王怒**：第 7 关解锁，全屏审判伤害，用来处理关键波次
- **功德牌**：结算不只是胜负，还给木、铜、银、金不同评价

### 战役规划：18 个主线关卡

| 阶段 | 关卡 | 主要学习点 | 章节 Boss |
|------|------|-----------|----------|
| 黄泉路 | 1-3 | 固定路线、建塔、清障、分裂怪、范围塔 | 黑无常 / 白无常 |
| 望乡台 | 4-6 | 招魂幡、魔抗、物理穿透、死亡沉默、分叉路线 | 牛头 / 马面 |
| 奈何桥 | 7-9 | 阎王怒、范围减速、隐身显形、怨气 | 孟婆 |
| 孽镜台 | 10-12 | 分身、镜面破碎、召唤精英 | 判官 |
| 轮回道 | 13-15 | 高物抗、高速、连续分裂、阴德规划 | 钟馗 |
| 阎王殿/轮回井 | 16-18 | 技能规划、限时无限波、最终三阶段 | 阎王 |

## 四、游戏素材生成

素材部分是 image2 参与度最高的环节之一。背景图、Boss 角色、敌人角色、技能图标、分享卡片，都可以用 image2 先生成候选，再由人筛选和修正，最后交给 Codex 接入资源路径和配置。

但**游戏素材和普通插画不一样，不能只看"好不好看"，还要看"能不能玩"**。

### 素材生产最关键规则

> **背景只承载地形、道路、建筑和氛围，不把塔位、敌人、障碍物、血条、UI 或攻击特效烘焙进背景图。**

### 背景图生成约束（给 image2 的硬性要求）

```
16:9，1280 x 720，Q 版可爱地府风。
只画背景地形、道路、建筑和氛围。
不要 UI，不要文字，不要敌人，不要炮塔，不要可清理障碍物。
不要八卦塔位、阴阳圆盘、放置环、血条、攻击范围。
道路必须从起点鬼门连续通向终点奈何灯。
主路两侧要保留可以放塔和障碍的陆地或平台。
装饰只能放在外围，不能像可点击对象或敌人。
```

### 素材分层

| 层级 | 内容 | 规则 |
|------|------|------|
| 背景层 | 地形、道路、桥、建筑、远景氛围 | 不烘焙可交互对象 |
| 运行时实体层 | 敌人、Boss、炮塔、障碍、掉落物 | 透明 PNG 或图集切片，由 Canvas 绘制 |
| 特效层 | 攻击、受击、状态、范围、预警 | 优先 Canvas 绘制，需要时独立贴图 |
| UI 层 | HUD、按钮、技能图标、弹窗、结算 | 不混入背景图 |

### Boss 设计方法

Boss 的 image2 提示词来自 boss-design.md，先写清楚机制，再反推视觉锚点：

- **黑无常**：黑衣、锁链、冷色魂火 → 玩法上负责眩晕和压迫
- **白无常**：白衣、招魂幡感、轻快轮廓 → 玩法上速度更快并带减速

### 素材生成和接入流程

1. 先写机制设计，明确素材服务什么玩法
2. 让 Codex 从设计文档里提取视觉关键词、尺寸、透明背景和禁画项
3. 用 image2 生成多张候选图
4. 人工筛掉不符合玩法可读性的图，比如道路断裂、把障碍画进背景、角色轮廓太复杂
5. 让 Codex 把选中的图接入 assets/，更新配置和云端 runtime 包
6. 进入游戏里看实际缩放后的效果，而不是只看原图

## 五、架构方案、设计方案生成

《地府小围城》的游戏代码架构不是 Harness，而是 TypeScript + Canvas 2D + 自研轻量运行时。整个项目可以拆成六块：

### 项目顶层框架

```
game-tower/
├── AGENTS.md                    # Codex 协作规则、关键路径、常用命令和验收规则
├── ARCHITECTURE.md              # 游戏代码架构、分层和数据流
├── docs/                       # Harness 文档工程：产品、设计、计划、质量、安全等
├── src/                        # 游戏运行时代码
├── content/                    # 数据配置：关卡、炮塔、敌人、障碍、图集
├── assets/                     # 原始和本地运行素材
├── cloud-assets/               # 云端素材说明和辅助目录
├── cloud-assets-upload/         # 待上传 OSS 的 runtime 素材包
├── browser/                    # 浏览器调试页面入口
├── minigame/                   # 微信小游戏工程输出目录
├── scripts/                    # 构建后同步、素材打包、平衡分析和模拟脚本
└── tests/                      # 视觉冒烟测试
```

### 第一块：Harness 文档工程

| docs/exec-plans/ | 进度、任务计划、验收清单、技术债 |
| docs/QUALITY_SCORE.md | 质量标准和交付口径 |
| docs/RELIABILITY.md | 资源失败、恢复、稳定性要求 |

### 第二块：游戏运行时代码

浏览器和微信小游戏共享同一套关卡配置、实体定义、游戏状态机和 Canvas 渲染器，只在平台能力上分叉：

| 层级 | 主要文件 | 职责 |
|------|---------|------|
| 浏览器入口 | src/main.ts、src/platform/browser.ts | 浏览器调试、DOM Canvas、输入、音频、本地存档 |
| 微信入口 | src/wechat-main.ts、src/platform/wechat.ts | 微信 Canvas、触摸、分享、微信音频和生命周期 |
| 云端资源 | src/platform/cloud-assets.ts | OSS runtime 素材基础路径和 URL 生成 |
| 状态层 | src/game/state.ts | 出怪、塔攻击、清障、阴德、轮回、胜负、结算 |
| 类型/schema | src/game/types.ts、src/game/level-schema.ts | 游戏类型、关卡配置结构和运行时字段 |
| 实体层 | src/entities/* | 敌人、炮塔、障碍、掉落物的运行时结构 |
| 渲染层 | src/ui/renderer.ts | 背景、敌人、炮塔、特效、HUD、弹窗 |
| 引擎工具 | src/engine/* | 主循环、输入、资源加载、路径采样、数学工具 |
| 音频 | src/audio/procedural-audio.ts | 浏览器端程序化攻击/UI/氛围音效 |

### 运行时流程

每帧大致是这个流程：

```
输入/触摸 -> GameState.handleTap()
资源检查 -> AssetStore 按需加载当前关卡素材
模拟推进 -> GameState.update(delta)
副作用消费 -> 音频事件、分享请求、进度保存
快照渲染 -> Renderer.render(snapshot)
```

### 运行时状态机

```
boot -> intro -> menu -> level_select -> loading_level
-> preparing -> playing -> paused -> victory / defeat -> settlement
```

这套状态机带来的好处很明显。后面接入剧情引导、关卡选择、通关解锁、失败复盘、暂停菜单、分享入口时，都不需要把逻辑硬塞进战斗中。

### 设计方案硬约束

- 设计坐标固定为 1280 x 720
- 普通关卡优先改 content 配置，不写关卡专属代码
- 新机制必须做成通用运行时能力
- 敌人必须先可见，再允许被塔攻击
- 移动角色要脸朝运动方向
- UI 面向手机横屏，不能遮挡右侧终点
